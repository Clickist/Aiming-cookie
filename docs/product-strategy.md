# 产品战略：两条产品线

## 核心决策

两条产品线，不混在一起。共享分析引擎层，只是数据采集层不同。

---

## 产品 A：轻量版（当前架构）

- 上传录像 → CV 分析 → 看结果
- 覆盖 tracking 场景（PTC 分析）
- 用户门槛最低，用来获客
- 免费或低价
- **链路**：打完 → 上传录像 → 看结果（3 步）

### 已知限制
- 纯录像无法检测 miss 的点击（画面零反馈）
- flicking 分析只能覆盖命中样本，有幸存者偏差
- 需要在 UI 里明确标注这个限制

---

## 产品 B：专业版（本地采集服务）

- 后台采集鼠标事件 + 屏幕录制 + 手部摄像头（可选）
- 覆盖 tracking + flicking + 手部分析
- 专业玩家/教练/战队，高价
- **链路**：启动采集服务 → 启动 KVK → 训练 → 看分析 → 看结果（5 步）
- 需要本地部署（Electron/Tauri），对电脑性能有要求

### 手部摄像头的价值

能提供的数据是其他方式拿不到的：
- 握鼠标时手指的紧张程度
- 手腕 vs 手臂的运动比例
- 点击时手指的发力模式
- 紧张时的微颤动

和 PTC（肌肉张力系数）有直接关联——如果能用摄像头验证"PTC 高的时候手确实在抖"，整个分析模型就有了物理验证。

### 手部数据的延伸价值

**运动科学分析**（市场上几乎没有竞品）：
- 疲劳检测——连续训练后手部微颤幅度增加，建议休息节奏
- 握姿识别——抓握/指握/掌握，不同握姿对应不同训练方法
- 发力模式——手腕主导 vs 手臂主导，和 DPI/灵敏度设置直接相关
- 肌肉代偿——异常动作模式可能说明某块肌肉过度紧张，给出拉伸/放松建议

**外设推荐 + 联盟佣金**（数据驱动的电商闭环）：
- 手大 + 掌握 → 推荐大鼠（G502、DA V3）
- 手小 + 指握 → 推荐小鼠（Viper Mini、GPX）
- 手腕主导 + 高 DPI → 推荐控制型鼠标垫
- 手臂主导 + 低 DPI → 推荐速度型鼠标垫
- 检测到握力过大 → 推荐轻量化鼠标 + 防滑贴

分析结果直接导向产品推荐，转化率高。用户看到"你的握力过大是因为鼠标太重"比看到广告有说服力得多。

---

## 为什么不能只做专业版

- 链路从 3 步变成 5 步，每多一步用户流失一批
- 后台采集对低配电脑有性能负担
- 手部摄像头需要额外硬件 + 摆放位置调试
- 轻量版是获客入口，专业版是变现手段

## 本地采集技术方案

```
后台采集服务（Electron/Tauri）
├── 鼠标事件 — pynput（事件驱动，几乎零开销）
├── 屏幕录制 — mss/DXGI 硬件编码（CPU <5%）
├── 手部摄像头（可选）— OpenCV + MediaPipe（30fps 降分辨率）
└── 时间戳对齐 + 数据打包 + 本地存储/上传
```

三者对游戏帧率影响估计 <3%。

---

## 三模块协同分析（优先级：低）

手部摄像头模块最终不应独立存在，而是与前两个模块协同，形成多模态诊断链：

```
Tracking 模块              Flicking 模块             手部模块
PTC、speed mismatch     减速段质量、过冲量       握姿、微颤、腕部偏转
        │                       │                        │
        └───────────┬───────────┘                        │
                    │                                    │
              游戏表现异常（"什么"）              姿态/发力原因（"为什么"）
                    │                                    │
                    └─────────────┬──────────────────────┘
                                  │
                          针对性建议（外设、灵敏度、握姿、休息）
```

### 协同价值

- **交叉验证**：PTC 飙高时手部摄像头是否同步检测到握力增大 → 验证而非推测
- **因果推断**：flick 过冲 → 同一时刻手部显示关节过度伸展 → "过冲是因为握力过大"
- **预测**：手部微颤幅度增加 → 接下来的 tracking/flicking 表现是否下降
- **闭环建议**：不仅说"你紧张了"，而是说"你紧张了，而且你的握姿是抓握，换个轻量化小鼠会缓解"

### 实现前提

三个模块各自独立可用后再做协同。手部模块依赖 MediaPipe Hands，需要用户额外摆放摄像头，门槛最高。优先级放在 tracking（已完成）→ flicking（进行中）→ 手部（规划中）之后。

## 架构分层

```
数据采集层（游戏无关）
├── 录像分析（CV）→ 目标位置、准星轨迹
├── 鼠标事件捕获 → 点击时间戳、准星移动轨迹
├── KovaaK's Stats CSV → 击杀时间戳、命中/miss、TTK、元数据
└── （可选）手部摄像头 → 手部姿态、握力分析

分析引擎层（场景无关）
├── tracking 分析 → PTC、speed mismatch
├── flicking 分析 → 反应时间、点击精度、减速段质量、miss 率
├── 手部分析 → 握姿识别、腕部偏转、微颤检测、运动模式
└── 协同分析 → 多模态交叉验证、因果推断、闭环建议

展示层
└── dashboard → 不同场景类型的标签页
```

数据采集层可以有多种来源，分析引擎层对数据格式做抽象，不绑死任何一种采集方式。这样以后加新游戏支持只需要加一个新的采集器，分析逻辑不用改。
